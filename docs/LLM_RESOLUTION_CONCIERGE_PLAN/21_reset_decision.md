# 21 Reset Decision

## Reset is not the default

Reset is **not recommended** from the observed repo state.

## Conditions where reset is unnecessary

- canonical grouping and artifact authority are intact
- route families and safety gates are observable
- link registry exists
- journey / task / rich menu structures exist
- the main gap is response assembly, not total architectural absence

## Conditions where reset might be reconsidered

- canonical owner groups become unusably inconsistent
- route boundaries become impossible to preserve additively
- existing task / journey / rich menu assets prove structurally incompatible with resolution-first composition
- safety cannot be preserved without deep route rewrite

## Current repo judgment

Observed repo state favors the smaller-than-reset path.

Why:

1. deterministic and audited assets already exist
2. the missing piece is a resolution assembly layer
3. reset would discard reusable task / link / rich menu infrastructure
4. add-only rollout with flags is more reversible than reset

## Why Concierge Layer + Task OS + Link-first is superior now

- it targets the observed failure mode directly
- it preserves current route and safety boundaries
- it reuses existing journey/task/rich menu data structures
- it allows shadow improvement before live expansion

## Decision

- reset: `not recommended for phase1`
- preferred path: `Resolution Concierge Layer + Link-first + Task OS bridge + Rich Menu bridge + Shadow loop design`
