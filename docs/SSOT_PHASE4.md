## P4-106: 運用判断フローの設計（Design Only）
### MUST
- 判断主体は必ず「人間（Admin）」である
- 判断は Yes / No / 保留 のみで表現する
- 判断結果はどこにも書き戻さない（記録しない）
- 判断結果による自動処理は行わない

### SHOULD
- 判断に使う情報は READ ONLY の一覧に限定する
- 判断不能ケースは「何もしない」を既定動作とする

### MUST NOT
- 自動判断・推薦・最適化を導入しない
- 既存フェーズの仕様や前提を変更しない

### 運用判断フロー（Who / When / What / Decision / Next）
| Who | When | What | Decision | Next |
| --- | --- | --- | --- | --- |
| Admin 運用者 | 定例（週次） | ユーザー状態一覧（memberNumber 有無 / checklist 完了数 / 最終アクション日時） | Yes/No/保留 | 何もしない（判断のみ） |
| Admin 運用者 | 定例（週次） | 通知状態一覧（sentAt / open 件数 / click 件数 / 最終反応日時） | Yes/No/保留 | 何もしない（判断のみ） |
| Admin 運用者 | 緊急時 | READ ONLY 集計・運用サマリー | Yes/No/保留 | 何もしない（判断のみ） |

### 判断できない場合の既定動作
- 情報不足で判断できない → 何もしない / 保留
- 矛盾があり判断できない → 何もしない / 差戻し
- 判断しないと決める → 何もしない / 保留

### 例外対応（設計のみ）
- 誤送信: Admin が事実を確認し、次の判断を保留する
- 想定外データ: Admin が判断不能として保留する
- 緊急停止: Admin が判断し、操作は行わない（設計のみ）

### Phase5 への引き渡し境界
#### Phase5 で実装してよいこと
- 人間判断の結果を安全に記録・参照できる仕組み
- 判断履歴を UI 上で確認できる範囲

#### Phase5 でも実装してはいけないこと
- 自動判断・自動送信
- AI 推薦・最適化
- 既存フェーズ仕様の変更
